# Event Triggers: File Watchers and Webhooks

Animus supports event-driven workflow dispatch in addition to manual invocation and
cron schedules. Two trigger mechanisms let external events kick off workflows
automatically: **file watchers** (ao-cli #211) that react to filesystem changes,
and **webhooks** (ao-cli #213) that accept inbound HTTP calls.

Both mechanisms produce a `SubjectDispatch` envelope just like any other trigger
source, so the same workflow definitions and execution rules apply regardless of
how a workflow was started. See [Subject Dispatch](../concepts/subject-dispatch.md)
for details on the dispatch contract.

---

## File Watchers

File watchers monitor paths inside your project directory and dispatch a workflow
whenever a matching file change is detected. This is useful for automating
responses to build artifacts, generated files, or content drops from other tools.

### Configuration

Declare file watchers in `.ao/triggers.yaml`:

```yaml
# .ao/triggers.yaml
triggers:
  - id: on-spec-change
    kind: file-watch
    watch:
      paths:
        - docs/spec/**/*.md
        - docs/spec/*.md
      events:
        - create
        - modify
    dispatch:
      workflow_ref: ao.task/standard
      subject:
        kind: ao.task
        id: "spec-review"
        title: "Review updated spec"
      vars:
        notify_reviewer: "true"
```

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | yes | Unique trigger identifier |
| `kind` | string | yes | Must be `file-watch` |
| `watch.paths` | string[] | yes | Glob patterns relative to the project root |
| `watch.events` | string[] | no | Event kinds to react to: `create`, `modify`, `delete`, `rename`. Defaults to `create` and `modify`. |
| `watch.debounce_ms` | integer | no | Milliseconds to wait after the last event before firing (default: `500`) |
| `dispatch.workflow_ref` | string | yes | Workflow to execute |
| `dispatch.subject` | SubjectRef | yes | Subject identity for the dispatch |
| `dispatch.vars` | map\<string, string\> | no | Variables passed to the workflow |
| `dispatch.priority` | string | no | Queue priority hint |

### Glob Pattern Rules

Paths are resolved relative to the project root (the directory containing
`.ao/`). Standard glob syntax applies:

- `*` matches any file name component, but not path separators
- `**` matches zero or more path components
- `?` matches a single character
- `{a,b}` matches either `a` or `b`

The daemon starts watching declared paths when it starts and reloads the trigger
config on `animus daemon reload`.

### Debounce Behavior

Rapid sequences of file events (for example, an editor saving multiple times in
quick succession) are collapsed into a single dispatch. The `debounce_ms` window
starts over each time a new event arrives. After the window expires with no new
events, the trigger fires once.

### Example: Auto-Review Generated Docs

Watch for documentation files generated by a build step and dispatch a review
workflow whenever they change:

```yaml
triggers:
  - id: generated-docs-review
    kind: file-watch
    watch:
      paths:
        - dist/docs/**/*.md
      events:
        - create
        - modify
      debounce_ms: 2000
    dispatch:
      workflow_ref: ao.task/standard
      subject:
        kind: custom
        id: "docs-review"
        title: "Review generated documentation"
```

## Webhooks

Webhooks expose an HTTP endpoint on the daemon's local port. An external
process — a CI job, another service, or a script — sends a POST request to
trigger a workflow. This bridges Animus into broader automation pipelines without
requiring the external caller to know about Animus internals.

### Enabling the Webhook Endpoint

The daemon's HTTP interface must be enabled. Set the listen address in
`.ao/config.yaml`:

```yaml
# .ao/config.yaml
daemon:
  http:
    listen: "127.0.0.1:8842"
    webhook_secret: "${AO_WEBHOOK_SECRET}"
```

`webhook_secret` is required when `listen` is set. The daemon validates each
inbound request with HMAC-SHA256 using this secret. Set the secret as an
environment variable to avoid storing it in plain text.

### Declaring Webhook Triggers

Declare webhook triggers in `.ao/triggers.yaml`:

```yaml
triggers:
  - id: on-ci-pass
    kind: webhook
    webhook:
      path: /hooks/ci-pass
      method: POST
    dispatch:
      workflow_ref: ao.task/standard
      subject:
        kind: ao.task
        id: "{{ payload.task_id }}"
        title: "{{ payload.task_title }}"
      vars:
        branch: "{{ payload.branch }}"
        commit_sha: "{{ payload.sha }}"
```

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | yes | Unique trigger identifier |
| `kind` | string | yes | Must be `webhook` |
| `webhook.path` | string | yes | URL path the daemon listens on (e.g., `/hooks/ci-pass`) |
| `webhook.method` | string | no | HTTP method (default: `POST`) |
| `dispatch.workflow_ref` | string | yes | Workflow to execute |
| `dispatch.subject` | SubjectRef | yes | Subject identity for the dispatch (supports `{{ payload.* }}` interpolation) |
| `dispatch.vars` | map\<string, string\> | no | Variables passed to the workflow (supports `{{ payload.* }}` interpolation) |
| `dispatch.priority` | string | no | Queue priority hint |

### Payload Interpolation

Field values in `dispatch.subject` and `dispatch.vars` can reference the
inbound JSON payload using `{{ payload.<key> }}` syntax. The daemon performs
this substitution before creating the dispatch entry. Missing keys are replaced
with an empty string.

### Calling the Endpoint

Send a `POST` request with a JSON body and the HMAC-SHA256 signature in the
`X-Animus-Signature` header:

```bash
# Compute signature
PAYLOAD='{"task_id":"TASK-099","task_title":"Deploy to staging","branch":"main","sha":"abc123"}'
SIG=$(echo -n "$PAYLOAD" | openssl dgst -sha256 -hmac "$AO_WEBHOOK_SECRET" | awk '{print $2}')

# Fire the webhook
curl -s -X POST http://127.0.0.1:8842/hooks/ci-pass \
  -H "Content-Type: application/json" \
  -H "X-Animus-Signature: sha256=$SIG" \
  -d "$PAYLOAD"
```

A successful request returns HTTP 202 and a JSON body confirming the dispatch:

```json
{
  "status": "accepted",
  "trigger_id": "on-ci-pass",
  "dispatch_id": "disp_01j9..."
}
```

### Security Notes

- Run the daemon listener on `127.0.0.1` only unless you have network-level
  controls in place. Binding to `0.0.0.0` exposes it to all interfaces.
- Rotate `AO_WEBHOOK_SECRET` the same way you rotate API keys.
- The HMAC signature check prevents forged payloads but does not encrypt the
  request body. Use a reverse proxy with TLS if the payload contains sensitive
  data.
- Replay attacks are not currently protected against. For high-security
  environments, include a nonce or timestamp in the payload and validate it
  inside your workflow.

### Example: Trigger on GitHub Actions Completion

A GitHub Actions workflow can call the daemon webhook at the end of a CI run:

```yaml
# .github/workflows/ci.yml (excerpt)
- name: Notify Animus daemon
  env:
    AO_WEBHOOK_SECRET: ${{ secrets.AO_WEBHOOK_SECRET }}
  run: |
    PAYLOAD=$(jq -n \
      --arg task_id "$TASK_ID" \
      --arg branch "$GITHUB_REF_NAME" \
      --arg sha "$GITHUB_SHA" \
      '{task_id: $task_id, branch: $branch, sha: $sha, task_title: "Post-CI review"}')
    SIG=$(echo -n "$PAYLOAD" | openssl dgst -sha256 -hmac "$AO_WEBHOOK_SECRET" | awk '{print $2}')
    curl -sf -X POST http://localhost:8842/hooks/ci-pass \
      -H "Content-Type: application/json" \
      -H "X-Animus-Signature: sha256=$SIG" \
      -d "$PAYLOAD"
```

## Trigger Engine

The trigger engine is the subsystem responsible for evaluating `.ao/triggers.yaml` and deciding whether an incoming event should produce a dispatch. In v40, the engine gained conditional filters, priority routing, dead-letter handling, and per-trigger statistics.

### Conditional Filters

Add a `filter` block to any trigger to gate dispatch on the event content:

```yaml
triggers:
  - id: on-main-pr
    kind: github
    github:
      installation: my-org
      repository: my-org/my-repo
      events:
        - pull_request.opened
        - pull_request.synchronize
    filter:
      # Only dispatch if the PR is not a draft
      expr: "!payload.pull_request.draft"
    dispatch:
      workflow_ref: ao.task/standard
      subject:
        kind: ao.task
        id: "pr-{{ payload.pull_request.number }}"
        title: "Review PR #{{ payload.pull_request.number }}"
```

The `filter.expr` field accepts a boolean expression using the same `{{ payload.* }}` namespace available in `dispatch` fields. Expressions are evaluated by the trigger engine before dispatch creation; events that do not satisfy the filter are counted as `filtered` in the trigger statistics but do not create a dispatch.

**Filter expression operators:**

| Operator | Example | Description |
|---|---|---|
| `!value` | `!payload.pull_request.draft` | Boolean negation |
| `a == b` | `payload.action == "opened"` | Equality |
| `a != b` | `payload.action != "closed"` | Inequality |
| `a && b` | `!draft && label == "ready"` | Logical AND |
| `a \|\| b` | `action == "opened" \|\| action == "reopened"` | Logical OR |
| `contains(a, b)` | `contains(payload.ref, "release")` | String containment |
| `startsWith(a, b)` | `startsWith(payload.ref, "refs/heads/main")` | String prefix |

Missing fields evaluate to `false` (not an error). Nest parentheses for complex expressions.

### Priority Routing

Set a `dispatch.priority` on any trigger to influence queue ordering:

```yaml
dispatch:
  workflow_ref: ao.task/standard
  subject:
    kind: ao.task
    id: "hotfix-{{ payload.issue.number }}"
    title: "Hotfix: {{ payload.issue.title }}"
  priority: high
```

| Priority | Description |
|---|---|
| `high` | Picked up before `normal` and `low` items |
| `normal` | Default; items are ordered by creation time within this band |
| `low` | Picked up only when no `high` or `normal` items are waiting |

Priority is advisory — the daemon processes one dispatch at a time per concurrency slot, so a `high` item does not interrupt an already-running `normal` run.

### Dead-Letter Queue

When a dispatch cannot be started after three consecutive attempts (for example, because the daemon restarted during startup), it moves to the dead-letter queue rather than being retried indefinitely.

View dead-letter items:

```bash
animus trigger dlq list
```

Re-queue a specific item:

```bash
animus trigger dlq requeue <dispatch_id>
```

Discard a dead-letter item:

```bash
animus trigger dlq discard <dispatch_id>
```

In the Animus Cloud dashboard, dead-letter items appear in the **Daemon → Queue** sub-tab under a **Dead Letter** section. Clicking a dead-letter item shows the original event payload and the error that prevented startup.

## Managing Triggers

### Listing Active Triggers

```bash
animus trigger list
```

Shows all configured triggers, their kind, current status (watching, paused, or error), and event counts since last reload.

### Pausing and Resuming

```bash
animus trigger pause on-spec-change
animus trigger resume on-spec-change
```

Paused triggers remain in the config but do not fire until resumed.

### Reloading Config

After editing `.ao/triggers.yaml`, reload the daemon without restarting it:

```bash
animus daemon reload
```

The daemon re-reads the trigger config and applies changes: new triggers start
watching or listening, removed triggers are stopped, modified triggers are
restarted.

### Trigger Statistics

View per-trigger counters since the last daemon restart:

```bash
animus trigger stats
```

Output:

```
ID                KIND        STATUS   FIRED   FILTERED   ERRORS   LAST FIRED
on-spec-change    file-watch  active   14      0          0        2m ago
on-ci-pass        webhook     active   7       1          0        18m ago
on-pr-opened      github      paused   0       0          0        —
```

| Column | Description |
|---|---|
| `FIRED` | Dispatches created by this trigger |
| `FILTERED` | Events that matched the trigger kind but failed the `filter.expr` check |
| `ERRORS` | Events that produced an internal error during evaluation |

Counters reset on `animus daemon reload`. Pass `--json` for machine-readable output.

## Trigger Sources in Dispatch

When a trigger fires, the resulting `SubjectDispatch` carries a `trigger_source`
field identifying the origin:

| Trigger Kind | `trigger_source` Value |
|---|---|
| File watcher | `file-watch` |
| Webhook | `webhook` |
| Cron schedule | `schedule` |
| Manual CLI | `manual` |
| Ready queue | `ready-queue` |
| MCP call | `mcp` |

Workflow phases can read `trigger_source` from the execution context to adjust
behavior — for example, skipping a notification step when a workflow is run
manually during development.

## Combining Triggers with Packs

Packs can ship trigger definitions alongside their workflow and schedule
definitions. Place trigger YAML at `triggers/triggers.yaml` inside the pack:

```text
my-pack/
  pack.toml
  workflows/
    respond-to-event.yaml
  triggers/
    triggers.yaml        # Pack-provided trigger definitions
  schedules/
    schedules.yaml
```

Project-level `.ao/triggers.yaml` entries take precedence over pack-provided
triggers. You can override a pack trigger by declaring a trigger with the same
`id` in your project config.
