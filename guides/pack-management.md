# Pack Management Guide

AO packs are installable plugin bundles that ship workflows, MCP server descriptors, runtime overlays, and domain behavior. This guide covers how to install, list, configure, and troubleshoot AO plugin packs.

## Understanding Packs

Packs extend AO's capabilities without modifying the core daemon. The daemon stays dumb—packs own the behavior. Each pack can provide:

- **Workflows**: Pack-qualified workflow refs like `ao.task/standard`
- **MCP Servers**: External tool integrations and namespaces
- **Runtime Overlays**: Agent and workflow runtime configuration
- **Subject Schemas**: Definitions for domain-specific subject types
- **Schedules**: Automated workflow triggers

AO supports three pack tiers:

| Tier | Description | Examples |
|------|-------------|----------|
| **Declarative** | Data-only: workflows, YAML config, schemas | Custom workflows, domain schemas |
| **Connector** | Declarative + MCP server integrations | Jira, Linear, GitHub integrations |
| **Native Module** | Compiled Rust modules linked via Cargo features | Custom subject resolvers, projectors |

For pack architecture details, see [Plugin Pack Kernel Architecture](../architecture/plugin-pack-kernel.md).

## Listing Packs

View all available packs (bundled and installed):

```bash
ao pack list
```

This displays pack IDs, versions, and source locations (bundled vs. installed).

To see detailed information about a specific pack:

```bash
ao pack inspect --pack-id ao.task
```

The inspect command shows:

- Pack manifest metadata (version, ownership, compatibility)
- Exported workflow refs
- Declared runtime requirements
- MCP server descriptors
- Subject kinds owned by the pack

### Pack Sources

Packs resolve from multiple sources in this order:

1. **Project-local overrides**: `.ao/plugins/<pack-id>/`
2. **Project-local workflows**: `.ao/workflows/`
3. **Installed packs**: `~/.ao/packs/<pack-id>/<version>/`
4. **Bundled packs**: Embedded in the AO binary

This precedence allows project-specific customizations without affecting global behavior.

## Installing Packs

### From a Local Path

Install a pack from a local directory:

```bash
ao pack install --path /path/to/pack
```

Add `--activate` to enable the pack immediately:

```bash
ao pack install --path /tmp/vendor.pack --activate
```

The pack is copied to `~/.ao/packs/<pack-id>/<version>/` and becomes available to all projects.

### Pack Requirements

Packs may declare runtime requirements such as:

- `node >=20`
- `python >=3.11`
- `uv`, `npm`, or `pnpm`

AO validates these requirements at install time. If a requirement is missing, the command fails with a clear diagnostic message.

To check requirements before installing:

```bash
ao pack inspect --pack-id vendor.pack
```

Look for the `[runtime_requirements]` section in the output.

## Configuring Packs

### Pinning Pack Versions

Projects should pin pack versions to ensure reproducible behavior. Pin a specific version:

```bash
ao pack pin --pack-id ao.task --version =0.1.0
```

Pin with a version constraint:

```bash
ao pack pin --pack-id vendor.pack --version ">=1.0.0,<2.0.0"
```

To use the bundled version of a first-party pack:

```bash
ao pack pin --pack-id ao.task --version builtin
```

### Pack Selection State

Project pack selections are stored in `.ao/state/pack-selection.v1.json`. This file tracks:

- Active pack IDs and pinned versions
- Enablement state for optional packs
- Pack-specific configuration overrides

**Do not edit this file directly.** Use `ao pack` commands to ensure consistency.

### Overriding Pack Content

For project-specific pack customizations, use the override directory:

```text
.ao/plugins/<pack-id>/
```

This directory structure mirrors the pack layout:

```text
.ao/plugins/ao.task/
  workflows/
    custom-task-workflow.yaml
  runtime/
    agent-runtime.overlay.yaml
```

Project overrides take precedence over installed and bundled pack content, allowing you to tailor behavior without forking the pack.

## Pack Structure

A well-formed pack has this filesystem shape:

```text
pack.toml                    # Pack manifest (required)
workflows/                   # Workflow YAML files
runtime/
  agent-runtime.overlay.yaml     # Agent runtime config overlay
  workflow-runtime.overlay.yaml  # Workflow runtime config overlay
mcp/
  servers.toml              # MCP server descriptors
  tools.toml                # MCP tool allowlists
schedules/
  schedules.yaml            # Automated schedule definitions
subjects/
  schemas.yaml              # Subject kind schemas
ui/
  labels.yaml               # UI labels and display metadata
```

### Pack Manifest (pack.toml)

The manifest defines pack identity, compatibility, and exports:

```toml
schema = "ao.pack.v1"
id = "ao.requirements"
version = "0.1.0"
kind = "domain-pack"
title = "AO Requirements"
description = "Planning and materialization flows for requirement subjects."

[ownership]
mode = "bundled"

[compatibility]
ao_core = ">=0.1.0"
workflow_schema = "v2"
subject_schema = "v2"

[subjects]
kinds = ["ao.requirement"]
default_kind = "ao.requirement"

[workflows]
root = "workflows"
exports = [
  "ao.requirements/draft",
  "ao.requirements/refine",
  "ao.requirements/execute",
]

[runtime]
agent_overlay = "runtime/agent-runtime.overlay.yaml"
workflow_overlay = "runtime/workflow-runtime.overlay.yaml"

[[runtime_requirements]]
kind = "node"
version = ">=20"

[mcp]
servers = "mcp/servers.toml"
tools = "mcp/tools.toml"
```

### Bundled First-Party Packs

AO ships with bundled packs that own core domain behavior:

| Pack ID | Description |
|---------|-------------|
| `ao.task` | Task lifecycle, standard delivery workflows, worktree adapters |
| `ao.requirement` | Requirement planning, refinement, materialization workflows |
| `ao.review` | Review and QA gate workflows, approval records |

Bundled pack manifests live in `crates/orchestrator-config/config/bundled-packs/`.

## Using Pack Workflows

Pack-qualified workflow refs follow the format `<pack-id>/<workflow-name>`:

```bash
ao workflow run --task-id TASK-001 --workflow-ref ao.task/standard
ao workflow run --task-id TASK-001 --workflow-ref ao.requirements/draft
```

Project YAML can wrap or override pack workflows:

```yaml
# .ao/workflows.yaml
workflows:
  standard-workflow:
    extends: ao.task/standard
    phases:
      implementation:
        timeout_secs: 7200  # Custom timeout
```

This approach keeps your project DRY while allowing targeted customizations.

## Searching for Packs

Search available packs by name or keyword:

```bash
ao pack search jira
ao pack search "issue tracker"
```

Search results show matching pack IDs, descriptions, and installation status.

## Troubleshooting

### Pack Not Found

**Symptoms**: `ao pack inspect` fails with "pack not found" or workflows fail to resolve.

**Steps**:

1. List available packs:
   ```bash
   ao pack list
   ```

2. Verify the pack ID and version:
   ```bash
   ao pack inspect --pack-id ao.task
   ```

3. Check project pack selection:
   ```bash
   cat .ao/state/pack-selection.v1.json
   ```

4. Ensure the pack is activated:
   ```bash
   ao pack install --path /path/to/pack --activate
   ```

### Missing Runtime Requirements

**Symptoms**: Pack installation fails with "missing runtime requirement" or MCP servers fail to start.

**Steps**:

1. Inspect pack requirements:
   ```bash
   ao pack inspect --pack-id vendor.pack
   ```

2. Install missing runtimes (examples):
   ```bash
   # Node.js
   brew install node
   
   # Python
   brew install python@3.11
   
   # uv
   curl -LsSf https://astral.sh/uv/install.sh | sh
   ```

3. Verify runtime versions:
   ```bash
   node --version
   python3 --version
   uv --version
   ```

4. Retry pack installation:
   ```bash
   ao pack install --path /path/to/pack --activate
   ```

### Workflow Resolution Fails

**Symptoms**: `ao workflow run` fails with "workflow ref not found".

**Steps**:

1. Verify the workflow ref format (must be pack-qualified):
   ```bash
   # Correct
   ao workflow run --task-id TASK-001 --workflow-ref ao.task/standard
   
   # Incorrect (unless you have a local workflow with this id)
   ao workflow run --task-id TASK-001 --workflow-ref standard
   ```

2. Check workflow exports from the pack:
   ```bash
   ao pack inspect --pack-id ao.task
   ```
   Look for the `[workflows.exports]` list.

3. Verify configuration precedence:
   ```bash
   ao workflow config get
   ```

### Pack Conflicts

**Symptoms**: Unexpected behavior after installing a new pack, or two packs define the same subject kind.

**Steps**:

1. List active packs:
   ```bash
   ao pack list
   ```

2. Check for subject kind conflicts:
   ```bash
   ao pack inspect --pack-id pack.one
   ao pack inspect --pack-id pack.two
   ```
   Look at `[subjects.kinds]` in both manifests.

3. Disable conflicting packs by removing from pack selection:
   ```bash
   # Edit .ao/state/pack-selection.v1.json to disable
   # Or reinstall without --activate
   ```

4. Use project overrides to resolve behavior:
   ```bash
   # Override in .ao/plugins/<pack-id>/
   ```

### MCP Server Fails to Start

**Symptoms**: Pack MCP servers fail with connection errors or timeout.

**Steps**:

1. Check MCP server descriptors:
   ```bash
   ao pack inspect --pack-id vendor.pack
   ```
   Look at `[mcp.servers]` configuration.

2. Verify required environment variables:
   ```bash
   # Example for Jira MCP server
   echo $JIRA_BASE_URL
   echo $JIRA_API_TOKEN
   ```

3. Test the MCP server command manually:
   ```bash
   npx -y @modelcontextprotocol/server-jira
   ```

4. Check daemon logs for MCP errors:
   ```bash
   ao daemon logs | grep -i mcp
   ```

5. Verify runtime dependencies:
   ```bash
   # Ensure Node.js is installed for npx-based servers
   which npx
   node --version
   ```

### Pack Override Not Taking Effect

**Symptoms**: Changes in `.ao/plugins/<pack-id>/` are ignored.

**Steps**:

1. Verify directory structure matches pack layout:
   ```bash
   ls -la .ao/plugins/ao.task/
   ```

2. Check file names and paths are correct:
   ```bash
   # Should be workflows/ not workflow/
   .ao/plugins/ao.task/workflows/custom.yaml
   ```

3. Validate YAML syntax:
   ```bash
   ao workflow config validate
   ```

4. Restart the daemon to reload configuration:
   ```bash
   ao daemon stop
   ao daemon start --autonomous
   ```

## Common Patterns

### Install and Pin a Third-Party Pack

```bash
ao pack install --path /path/to/jira.pack --activate
ao pack pin --pack-id jira.issue --version =0.3.0
ao workflow run --task-id TASK-001 --workflow-ref jira.issue/sync
```

### Override a Bundled Pack Workflow

Create `.ao/plugins/ao.task/workflows/custom-standard.yaml`:

```yaml
schema: ao.workflow.v2
id: ao.task/custom-standard
title: Custom Standard Workflow
extends: ao.task/standard

phases:
  implementation:
    timeout_secs: 14400  # 4 hours instead of default
```

Then use the custom ref:

```bash
ao workflow run --task-id TASK-001 --workflow-ref ao.task/custom-standard
```

### Use Multiple Pack Versions Across Projects

Each project maintains its own pack selection:

```bash
# Project A
cd ~/projects/project-a
ao pack pin --pack-id ao.task --version =0.1.0

# Project B
cd ~/projects/project-b
ao pack pin --pack-id ao.task --version =0.2.0
```

The daemon respects project-local pack selection when executing workflows.
